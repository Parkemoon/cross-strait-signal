# AI Pipeline + Processors

## Three-tier AI pipeline (`ai_pipeline.py`)

- **Tier 1**: Gemini 3.1 Flash Lite вҖ” classifies all pre-filtered articles (batch limit: 500); `temperature=0.1`, `thinking_level=medium`.
- **Tier 1 runs through the Gemini Batch API by default** (`run_tier1`, code-review В§3.5 вҖ” batch tokens bill ~50% of interactive, and Tier 1 is the largest line on the bill). Per pipeline tick: collect any job submitted on a previous tick вҶ’ submit the current backlog as one job (`gemini_batch_jobs` table tracks state; articles inside a `submitted` job are excluded from re-selection) вҶ’ poll up to `TIER1_BATCH_WAIT_MIN` (default 20) minutes so small jobs apply same-tick. Failed/expired/48h-stale jobs release their articles automatically. Both modes share `_tier1_prompt` + `_apply_tier1_analysis`, so requests and post-processing are identical; the conditional Tier-2 escalation review stays interactive. `GEMINI_TIER1_MODE=interactive` in `.env` restores the sequential path, and batch submission errors fall back to it automatically. Batch usage rows log stage `tier1_batch` with model `<model>@batch` so `usage_report.py` prices them at the batch rate.
- **Tier 2**: Gemini 3.5 Flash вҖ” re-reviews only escalation-flagged articles; same temperature. The Tier-2 prompt reuses `ANALYSIS_SYSTEM_PROMPT` (so sentiment is judged by identical rules вҖ” no drift), but appends `_ESCALATION_REVIEW_DIRECTIVE` telling the model to populate only the sentiment/escalation/topic/confidence fields and return empty extraction arrays (it re-judges, it doesn't re-extract). The caller reads back only those fields.
- **Tier 3**: Human review queue вҖ” articles where Tier 1 and Tier 2 disagree; stay hidden from dashboard until resolved.
- **Age filter**: `process_unanalysed_articles` only processes articles with `published_at >= datetime('now', '-180 days')` вҖ” old DB backlog never reaches the AI pipeline.
- **Thinking levels**: Tier 1 + `diplomacy_only` are `medium`; the template-following `exercise_only` + `poll_only` passes are `low` (thinking tokens bill at the output rate and dwarfed the useful output at `medium`). `social` translation is `low`.
- **Officials roster injection**: the CURRENT roster (`_OFFICIALS_CURRENT_BLOCK`, ~29 rows) is static and injected into every Tier-1/Tier-2 prompt (kept in the cacheable prefix); the ~99 FORMER officials are filtered per-article by name presence (`_officials_former_block`, same idea as the dynamic glossary) instead of shipping the whole list every call.
- **Transient-error handling**: the Tier-1 per-article `except` uses `_is_transient_error` вҖ” rate-limit (429), 5xx, and network/timeout errors leave `ai_processed=0` (rolled back, retried next run) instead of tombstoning the article as `ai_processed=1`. Only genuine parse/validation failures tombstone (so a pathological article can't retry forever). There is no automatic backoff/retry within a run.

## Dynamic glossary injection (`glossary.json`)

Loaded once at module level. Before each API call, both the article title and body text are scanned (`generate_dynamic_glossary(content, title)`) and matching terms (politicians, military assets, institutions in both Simplified and Traditional Chinese) are injected as a `CRITICAL TERMINOLOGY MAPPING` block to prevent romanisation hallucinations. Add new terms to `glossary.json` without touching Python. Always add both Traditional and Simplified Chinese forms for the same term.

## Entity canonical normalisation (`entity_canonical.json`)

Applied *after* AI extraction to normalise `name_en` on entity rows. Distinct from `glossary.json` (which is injected into the prompt *before* analysis). Covers parties, PLA branches, theater commands, and institutions in addition to named individuals. Since 2026-07-10 (code-review В§4.6) resolution lives in **`shared/entity_norm.py`** вҖ” ONE resolver used by both the pipeline write path (`_normalise_entity_name`) and the back-fill (`scripts/renormalise_entities.py`), so a canonical edit can always repair historical rows. The JSON has three tables:

- **`canonical`** (zh вҶ’ en): exact match, always wins first. Single-char keys are allowed (an explicit `зҝ’` вҶ’ Xi Jinping is a deliberate call); the other tiers require 2+ chars.
- **`title_tokens`**: explicit role words (йғЁй•·, еёӮй•·, дё»е§”, дё»еёӯ, зҷјиЁҖдәә, вҖҰ). If a token appears in an unresolved name, the part *after* it (title-first: `еңӢйҳІйғЁй•·йЎ§з«Ӣйӣ„` вҶ’ йЎ§з«Ӣйӣ„) or вҖ” only when the token is the ending вҖ” *before* it (name-first: `иіҙжё…еҫ·зёҪзөұ`) is exact-matched against `canonical`. High precision by construction: the remainder must be an exact canonical key. (The ending-only guard on the name-first side stops `зёҪзөұеәңз§ҳжӣёй•·жҪҳеӯҹе®ү` from head-matching зёҪзөұеәң.)
- **`fold_prefixes`**: opt-in canonical keys allowed to prefix-match longer extracted forms (`жјўе…ү41иҷҹжј”зҝ’` вҶ’ жјўе…ү41, `и§Јж”ҫи»Қжө·и»ҚйҷёжҲ°йҡҠ` вҶ’ и§Јж”ҫи»Қжө·и»Қ вҖ” longest listed prefix wins). Only listed keys participate. This replaced the old open-ended bidirectional prefix scan, which corrupted the corpus at scale: `дёӯиҸҜж°‘еңӢ` вҶ’ "Republic of China **Armed Forces**", `йҹ“еңӢ` вҶ’ "**Han Kuo-yu**", `зҰҸе»ә` вҶ’ the **carrier**, `еңӢйҳІйғЁй•·йЎ§з«Ӣйӣ„` вҶ’ the **ministry**. Don't fold keys whose compounds are heterogeneous вҖ” `дёӯе…ұ` was removed from the list because `дёӯе…ұи»ҚиүҰ` (warships) folded to the party; its exact entry stays. Same trap: `дёӯе…ұдёӯеӨ®еҸ°иҫҰ` has an explicit TAO entry so it can't fold to CCP.
- Editing this JSON only affects **future** extractions. To fix rows already in the DB, run `scripts/renormalise_entities.py --apply` вҖ” since the resolver is shared and every tier is explicit, the back-fill is safe on historical rows (it repaired ~6.3k staging rows on 2026-07-10, including the old scan's mislabels). A legacy flat `{zh: en}` file still loads (treated as canonical-only).
- When adding a person to `glossary.json`, add the same entry here too вҖ” otherwise the AI may translate their name correctly but store it under a non-canonical romanisation in the entities table.

## Key figure statement extraction (`key_figures.json`)

Tier 1 also extracts attributed `(speaker, statement)` pairs into the `key_figure_statements` table as `pending` candidates. The curated figure list lives in `key_figures.json` вҖ” 10 figures with Chinese/English names, roles, party field (DPP/KMT/PRC), portrait filenames, and alias lists used for speakerвҶ’figure_id matching. Tier 2 does NOT re-insert statements (only Tier 1 writes to this table). Statements require analyst approval via the Key Figures panel before appearing on the dashboard вҖ” intentional to prevent misattribution.

## Military exercise extraction

When Tier 1 classifies an article as `MIL_EXERCISE`, it side-extracts up to a handful of exercise candidates (name_en, name_zh, performer вҲҲ {PRC, ROC, US, JP, MULTI}, kind, start/end date, location label + best-effort lat/lng, English description) into `military_exercises` with `status='pending'`. The published year of the article is passed in as an anchor so partial date strings ("Aug 19") resolve to the right year. Pending rows are hidden from `/api/military/exercises` until an analyst approves them вҖ” see [[api-routes]] for the editorial flow and the canonical-name auto-merge. Extracted dates that fall outside Вұ1 year of `published_at` are silently dropped (likely a hallucination).

A second pass (**Step 3b** in `run_pipeline.py`, `process_exercise_only_articles`) runs the same extraction against military-source articles (YDN) the keyword pre-filter rejected. No `ai_analysis` row is written for these вҖ” they exist only to feed the exercise tracker. Capped at 30 per run, last 14 days. Idempotency is via `articles.exercise_scanned_at`, stamped after every scan **including zero-yield** (before the marker, no-yield articles re-qualified every 6h tick вҖ” 70% of the stage's Gemini calls were repeat scans). Transient API errors (429/5xx/network, `_is_transient_error`) skip the stamp so the article retries next tick; parse failures stamp it so a pathological article can't retry forever. See `.claude/rules/scrapers.md` for the geocoding sidecars.

## Poll extraction

Tier 1 side-extracts public-opinion polls (TW identity / unification / approval / vote intent) into the `polls` table as `approval_status='pending'`. Same editorial-gate pattern as `military_exercises`. The prompt requires four signals before extracting вҖ” named pollster, fielded date, sample size, and at least one numeric option вҖ” so passing references to historical poll numbers don't pollute the queue.

Two design quirks worth knowing:

- **`pollster_hint` is free text.** The AI copies the organisation name verbatim from the article. `_resolve_pollster_id` then maps it to a `pollster_id` via a three-layer match (exact вҶ’ name contains hint вҶ’ hint contains name) against a lookup of slug/name_zh/name_en, falling back to the seeded `unknown` pollster when nothing matches. The roster is loaded once per pipeline run. There's also a layer-0 glossary fallback: if the hint contains a `_MASTER_GLOSSARY` key, the glossary's English value is added as a second lookup candidate. This is what makes MAC's short forms (`йҷёе§”жңғ`, `йҷҶе§”дјҡ`) resolve correctly even though they aren't contiguous substrings of the formal `еӨ§йҷёе§”е“Ўжңғ` вҖ” the glossary already canonicalises them to "Mainland Affairs Council (MAC)", and the existing layer-2 substring match catches `Mainland Affairs Council` inside that string. Free alias resolution for any pollster whose Chinese variants are already in the glossary; no separate alias table needed.
- **`pending_results_json` stages the questions until approval.** `poll_results.question_id` is a NOT NULL FK to `poll_questions`, but `question_key` is analyst-assigned (never AI-extracted) so long-tail miscategorisation can't corrupt cross-pollster trend charts. To bridge the gap, the extracted `{questions:[{question_text_zh, question_text_en, family_hint, options:[{label_zh, label_en, percentage, option_order}]}]}` is held in `polls.pending_results_json` while the row is `pending`. On approve, the review queue picks a `question_key` per question and the server materialises `poll_results` rows from the JSON then NULLs the column.

The extraction validates that `fielded_start` matches `YYYY-MM-DD`, drops options whose percentage isn't a 0вҖ“100 float, and drops questions left with no usable options. Date anchoring follows the same rule as military exercises вҖ” partial dates resolve against the article's published year.

Four prompt-level rules to know about when tuning extraction quality:

- **Multi-question extraction is mandatory.** The prompt asks for ALL distinct questions in a poll write-up (TVBS PDFs especially carry 4вҖ“6 questions in one survey wave) and allows the model to synthesise a short `question_text_zh` when the article presents only a topic header + option breakdown. An earlier "false negatives strongly preferred" framing caused cherry-picking вҖ” usually the headline question got dropped in favour of a subgroup factoid. If you're touching the Tier 1 or Step 3c poll prompt, keep both the "extract ALL distinct questions" instruction and the worked multi-question example; the subgroup-cross-tab skip rule is what prevents false positives, not the cherry-pick framing.
- **Aggregate-vs-intensity rule** вҖ” for ANY binary-with-intensity scale (satisfied/dissatisfied, trust/distrust, favourable/unfavourable, agree/disagree, good/bad вҖ” i.e. a question collapsing to two directional positions plus no-opinion), the prompt instructs the model to ALWAYS extract the 3-option aggregate form (positive / negative / no-opinion). When the article reports both top-line aggregate and parenthetical intensity breakdown, use the aggregate verbatim. When the article reports ONLY the breakdown, compute the aggregate by summing the two positive intensities and the two negative intensities вҖ” consistency across waves outranks preserving journalistic detail. EXCEPTION (preserve all options): multi-option position scales like the зөұзҚЁ 7-step scale, statement-list questions where each option is a distinct claim being rated, multi-candidate vote-intent, and ranking questions вҖ” these have more than two underlying positions. **Note the "agree/disagree" inclusion:** an earlier draft of this rule treated agree/disagree as a "true Likert" exception, but the structural pattern is identical to the others (very/somewhat agree, very/somewhat disagree, no opinion) and a re-extraction test on 2026-05 surfaced the inconsistency. The 2026-05 My-Formosa wave was the canonical incident: 11 questions stored disaggregated, breaking trend charts under their existing question_keys; fixed by reaggregating + this prompt rule.
- **Canonical option labels** вҖ” the prompt's CANONICAL NO-OPINION + VOTE-INTENT blocks specify exact `label_zh` / `label_en` strings the model must emit for "didn't answer" buckets and vote-intent residuals (`жңӘжҳҺзўәеӣһзӯ”` / "No response" for the general case; `е°ҡжңӘжұәе®ҡ` / "Undecided" and `дёҚжҠ•зҘЁжҲ–жҠ•е»ўзҘЁ` / "Won't vote / Spoiled ballot" as the two distinct vote-intent residuals; candidate names verbatim without party prefix). The 2026-05 My-Formosa wave revealed 6 different EN translations of the same `жңӘжҳҺзўәеӣһзӯ”` across waves and 5 inconsistent vote-intent label formats вҖ” each variant became a separate trend line. The prompt is the first defence; `scripts/canonicalise_poll_labels.py` (config: `scraper/processors/poll_labels_canonical.json`) is the second вҖ” re-run on demand to catch any historical drift or future model regression. Edit the JSON when a new variant appears; no code change needed.
- **`sources.is_pollster_direct` auto-populates `source_url`.** When the article's source row carries this flag (set by `seed_sources.py` вҖ” TVBS Poll Center, My-Formosa, ETtoday Polls), `_insert_poll_row` sets `polls.source_url = article.url` because the article *is* the pollster's own release page вҖ” saves the analyst a manual paste in the review queue. Set the flag on a source's seed entry when a new pollster homepage gets scraped directly (rather than reached via news coverage). (Was a hardcoded display-name set in `ai_pipeline.py` until 2026-07-08 вҖ” the flag now travels with the row, so renames can't silently break the auto-populate.)

A second pass (**Step 3c** in `run_pipeline.py`, `process_poll_only_articles`) runs a stripped poll-only prompt against TW-side articles the keyword pre-filter rejected, restricted to titles containing ж°‘иӘҝ or ж°‘ж„ҸиӘҝжҹҘ. This catches Lai-approval / vote-intention / identity polls whose write-ups lack a cross-strait keyword angle вҖ” the keyword filter correctly rejects them for the main feed, but the polling tracker needs them. Title-only trigger is high-precision: articles whose title carries ж°‘иӘҝ are almost always primarily about a poll. The pass is idempotent via `articles.poll_scanned_at` вҖ” stamped after every scan including zero-yield (before the marker, no-yield articles re-qualified every tick: 96% of the stage's calls were repeat scans, one article scanned 56Г—), with the same transient-vs-permanent error handling as Step 3b вҖ” capped at 30 per run over a 14-day window, and shares the `_insert_poll_row` helper with the main loop so validation stays consistent. The prompt explicitly skips party primary-selection polls (еҲқйҒёж°‘иӘҝ) and polls of non-TW/PRC publics, the two noise classes the title trigger most often catches.

## Diplomacy stance extraction (Phase 2c)

Tier 1 also side-extracts third-country stances on the Taiwan / cross-strait question into `diplomacy_statements` as `approval_status='pending'` вҖ” a SEPARATE axis from the core sentiment instrument, which deliberately discards third-party interactions ("third-party interactions are NOT cross-strait signals"); this captures exactly that discarded signal. Same editorial-gate pattern as `military_exercises` / polls. `scripts/backfill_diplomacy_statements.py` shares the same `_insert_diplomacy_row` helper (org-exclusion + validation identical to the live pass); flags `--days` / `--limit` / `--topics` / `--all-topics` / `--db` / `--dry-run`.

Each extracted statement carries `(country, country_iso, speaker, authority_tier, stance вҲ’1..+1, statement_en, statement_zh, stated_date)`. `_insert_diplomacy_row` drops rows that don't resolve to a third country or carry no numeric stance.

- **Country resolution (`_resolve_country_iso`, config `country_iso.json`)**: trusts a valid 2-letter ISO, else falls back to name aliases; accepts plausible OFF-roster 2-letter codes so long-tail countries still reach the review queue. `exclude_iso` drops the cross-strait principals (CN/TW/HK/MO) **and `UN`**. International ORGANISATIONS are rejected by `_looks_like_org` (name patterns вҖ” "organisation", "United Nations", "civil aviation", WHO/WTO/ICAO/NATO/вҖҰ acronyms) because this axis is third-COUNTRY stance; the one deliberate exception is the **EU bloc** (`EU` pseudo-code). The org guard was added after ICAO leaked in under an invented `UN` code from a pundit's rhetorical "even ICAO can't stand it". **De-facto states**: `iso_override_by_name` in `country_iso.json` makes the emitted NAME outrank the emitted code (`XS` Somaliland, `XK` Kosovo) вҖ” the model reliably stamps the neighbouring recognised state's code (Somaliland вҶ’ `SO` = Somalia), which painted the wrong polygon on the diplomacy map until 2026-07-10. Keep it in sync with `DEFACTO_ISO` in `scripts/build_world_geojson.py`, which stamps the same codes onto the basemap features NE leaves unkeyed.
- **`stance` вҶ’ `stance_label`** via `_stance_label` (bands at Вұ0.2 / Вұ0.6). Mirrored in `api/routes/diplomacy.py` вҖ” keep thresholds in sync. `source_side` (`_source_side_from_place`): TW вҶ’ TW; PRC/HK/MO вҶ’ PRC; else INTL вҖ” the reporting outlet's side, not the speaker's.
- **CJK guard** on `statement_en`: if >15% CJK it's dropped to NULL (don't reject the row); `stated_date` must match `YYYY-MM-DD` else NULL.
- **Scope gate + sign guard (prompt-level).** Stance scoring runs on `gemini-3.1-flash-lite` (baked into the Tier-1 call, so it shares Tier 1's model вҖ” you can't bump it without bumping all of Tier 1). Flash-lite has two failure modes on this axis: (1) **scope creep** вҖ” it pulls in anti-PRC / pro-PRC content with no Taiwan nexus (Xinjiang/Tibet human-rights critiques, WWII anti-militarism, freedom-of-navigation boilerplate) and scores it directional; (2) **toneвҶ’sign leak** вҖ” because the scale is sign-inverted (negative = pro-Beijing), a harshly-worded *condemnation of Beijing* gets mapped to a *negative* stance, the opposite of intent. The prompt's **SCOPE GATE** ("no explicit Taiwan/cross-strait/one-China nexus вҶ’ extract nothing"), **SIGN IS DIRECTION NOT TONE** guard, and four **WORKED EXAMPLES** target both. These live in the shared `_DIPLOMACY_RULES` constant, concatenated into both the `DIPLOMACY STATEMENTS` bullet of `ANALYSIS_SYSTEM_PROMPT` (live path) and `_DIPLOMACY_ONLY_PROMPT` (backfill) вҖ” edit the constant, both prompts pick it up. (The named-exercise roster is likewise shared as `_NAMED_EXERCISES` across the Tier-1 bullet, `_EXERCISE_ONLY_PROMPT`, and `scripts/backfill_military_exercises.py`.) A 2026-07 blind 3.5-flash audit (150-row sample) measured the base rates: sign flips ~1%, off-axis over-extraction ~11% (skewing pro-Taipei). A two-pass off-axis detector (detect вҶ’ conservative confirm, sole-statement countries held back) dismissed 98 confirmed off-axis approved rows on prod (`reviewed_by='offaxis-audit-20260701'`, reversible). Re-running that detector periodically is a candidate maintenance step.

## Relevance gate

The prompt requires the model to set `is_cross_strait_primary` (bool) as its first decision before classification. If false, `topic_primary` is forced to `NOT_RELEVANT` both by the model and by a Python-level enforcement check. `NOT_RELEVANT` is a special pseudo-topic that exists in the DB but is not part of the 28 visible categories вҖ” it marks filtered articles and is never shown in the UI. PRC sources writing about Taiwan are explicitly exempt вҖ” their cultural/lifestyle coverage of Taiwan is analytically relevant (POL_TONGDU framing) and should not be filtered.

## Sentiment consistency check (`_validate_sentiment()`)

Called after each Tier 1 and Tier 2 extraction. Flags label/score band mismatches (e.g. `hostile` with score > вҲ’0.3) and directional labels with empty `sentiment_reasoning` to the human review queue (`needs_human_review=1`). Reuses the same low-confidence flag path вҖ” review reasons are concatenated with ` | `.

## Keyword pre-filter (`keyword_filter.py`)

Directional logic:
- PRC/HK/SG sources: must mention Taiwan, ROC, or relevant territories.
- Taiwan sources: must mention PRC, mainland, Hong Kong, or Macau.

Only `title + content[:2000]` is checked вҖ” full content is not used, to prevent page navigation/sidebar cruft from passing irrelevant articles. Irrelevant articles are marked `ai_processed=1` and skipped вҖ” they never reach the AI API.

## Social Pulse translator (`social_translator.py`)

Separate lightweight pipeline for social data вҖ” does NOT go through the article AI pipeline. Batch-translates `social_pulse` rows where `title_en IS NULL` using Gemini 3.1 Flash Lite (`thinking_level=low`). Runs as Step 2b in `run_pipeline.py` after the social scrapers.

## Token usage logging (`scraper/utils/usage_log.py`)

Every Gemini call routes its `usage_metadata` through `log_usage(stage, model, response, article_id=None)`, which appends one JSON line per call (prompt / cached / thoughts / output / total tokens) to `$GEMINI_USAGE_LOG` (default `/var/log/gemini-usage.jsonl`). Calls are tagged by stage вҖ” the article-AI sites (`tier1`, `tier2`, `exercise_only`, `poll_only`, `diplomacy_only`) plus the social translator (`social`). Append-only, no schema; the logger swallows all errors so instrumentation can never break the pipeline. Aggregate with `scripts/usage_report.py` (`--days`, `--by stage|model|day`). Use it to attribute the bill by stage before tuning cost (context caching / `MAX_PROMPT_CONTENT_CHARS` / thinking level).

## Don't reference `test_ai.py`

`test_ai.py` is a legacy prototype script вҖ” do not use it as a reference. It uses a stale prompt with old topic codes (`POL_UNIFICATION`, `POL_DOMESTIC`) and old sentiment values (`escalatory`/`conciliatory`) that no longer match the DB schema or the real pipeline in `ai_pipeline.py`.
